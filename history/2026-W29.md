### [code] lib/src/organisms/ui_adaptive_nav_shell.dart · test/ · docs/ · .claude/skills/flutter-adaptive-navigation/ — adaptive navigation shell (v0.5.0)
- **What:** Harvested an external "Flutter Complete Adaptive Layout Guide" into kit-conformant form
  (docs + component + skill), reconciling its ideas against the kit's rules instead of adopting it
  verbatim:
  - **New organism `UiAdaptiveNavShell`** + value type `UiNavDestination`
    (`lib/src/organisms/ui_adaptive_nav_shell.dart`): a top-level nav scaffold that switches pattern by
    *available pane width* via `UiBreakpoints.classify` inside a `LayoutBuilder` — M3 `NavigationBar`
    for `compact`, compact `NavigationRail` (`extended:false`, `labelType:selected`) for `medium`,
    extended rail for `expanded`/`large`. Controlled selection (`selectedIndex`/`onDestinationSelected`);
    all-optional/additive API. Token-only, zero added deps. Handles the Flutter assertion that an
    extended rail requires `labelType == null` (labels render inline).
  - **Test** `test/ui_adaptive_nav_shell_test.dart` — 8 tests across the four width tiers (driven via
    `setSurfaceSize`, since a plain `SizedBox` clamps to the 800px default surface), rail
    extended/labelType per tier, bottom-bar + rail selection callbacks, body render, and the
    ≥2-destinations assertion.
  - **Skill** `.claude/skills/flutter-adaptive-navigation/SKILL.md` — when to reach for
    `UiAdaptiveNavShell` vs `UiResponsive`/`UiBreakpoints`, the tier→nav mapping, the rejected-patterns
    list, and the why-behind-the-rules (so future changes can override deliberately).
  - **Docs:** `docs/flutter-adaptive-ui-design-specification.md` v3.0→3.1 (new §3.4 shell recipe, §3.5
    reconciliation table vs. the external guide, §9 Future Extensibility); `design-system-contract.md`
    v1.4→1.5 (organism inventory + revision row); `CHANGELOG.md` 0.5.0; `pubspec.yaml` 0.4.0→0.5.0;
    barrel export + doc-comment.
- **Why:** User asked to integrate the external guide. It can't be adopted as-is — it adds four
  runtime packages (`window_manager`/`flutter_adaptive_kit`/`universal_breakpoints`/`fluent_ui`, vs.
  the zero-dependency rule), uses 600/900/1200 (kit uses centralized 600/840/1200), M2
  `BottomNavigationBar` (kit is M3-only), and `Platform.isWindows`-driven layout (kit decides by width,
  not OS). Its four tiers map cleanly onto `UiDeviceClass`, so the kit provides the *shell*; window
  management, keyboard shortcuts, and `go_router` state stay in the consuming app (separation of
  concerns). Additive-only → MINOR bump; consumers pinned to v0.4.0 unaffected until they bump.
- **Refs:** plan `please-review-the-guide-serialized-coral.md` (user-reviewed across two rounds — a
  future-proofness review confirmed the kit approach is already superior/2025-26-aligned and rejected
  adding speculative `desktopNavType`/`breakpoints` params now, keeping the API additive for later).
  Design rationale persisted to memory `adaptive-nav-shell-design`. Verified: `flutter analyze` clean,
  `flutter test` all 102 green.
