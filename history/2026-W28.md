### [code] lib/src/components/ · lib/src/theme/ui_tone.dart · lib/src/theme/ui_tuning.dart — common-atoms completion (v0.2.0)
- **What:** Built the standard Material atom set on top of the six existing core atoms, all following
  the established conventions (StatelessWidget, one `Ui<Name>` per file, token-only, `onChanged: null`
  = disabled, optional `tooltip`, default-with-override height via `UiTuning`, mirrored tests, barrel
  exports):
  - **Form-input:** `ui_radio.dart` (`UiRadio<T>` + `UiRadioGroup<T>` on Material 3's `RadioGroup`
    ancestor API — the deprecated `groupValue`/`onChanged`-on-`Radio` shape is intentionally avoided;
    disabled group = no-op `onChanged` + `IgnorePointer`/`Opacity`), `ui_switch.dart`, `ui_slider.dart`
    (no height override — a `Slider`'s ~48px thumb overlay would clip in the 35px control row).
  - **Buttons:** `ui_icon_button.dart` (standard/filled/outlined variants + semantic `tone`).
  - **Layout/text:** `ui_card.dart` (surface, optional `onTap`), `ui_text.dart` (typography atom with
    title/body/label/caption/mono roles).
  - **Feedback:** `ui_progress_indicator.dart` (circular/linear, determinate/indeterminate),
    `ui_avatar.dart` (initials/icon/image, colorScheme-tinted), `ui_chip.dart` (generic input/filter/
    choice chip, distinct from `UiStatusChip`; height fixed by sizing the *label* + zeroing the chip's
    vertical padding + `shrinkWrap`, per the chip-height trap in the component skill).
  - **New token** `theme/ui_tone.dart` (`UiTone` enum + `foreground()` extension) so `UiIconButton`
    reads tone from the **theme layer**, not by importing `UiButton`'s `UiButtonTone` (which would
    violate the component→theme-only import boundary). `UiButtonTone` left intact for API stability.
  - `UiTuning`: added `radioHeight` / `switchHeight` (triple pattern, seeded from `UiSizing.controlHeight`,
    wired into `reset()`); `UiChip` reuses the existing `chipHeight`.
  - Refactored `UnderMaintenancePage` onto `UiCard` (composite→component, allowed).
  - Docs/versioning: barrel doc-comment, `CHANGELOG.md` 0.2.0, `pubspec.yaml` 0.1.0→0.2.0,
    `design-system-contract.md` v1.1→1.2 (atom inventory note), `ROADMAP.md` (Epic 1 status-chip test
    ticked, new completed Epic 4, Icebox reworded). Also `test/ui_status_chip_test.dart` (Epic 1 gap).
- **Why:** User explicitly asked to "design the most common atoms UI first." Button/text-field/dropdown
  already existed, so this fills the rest of the Material atom set. This is a **deliberate override** of
  the roadmap's "don't build speculatively" stance (which named radio group / segmented control /
  tooltip as wait-conditions) — reconciled by moving the built atoms into a real completed epic rather
  than leaving the roadmap self-contradictory. Additive-only → MINOR bump; consumers pinned to `v0.1.0`
